# TradeHub

> 상품 등록, 예약, 채팅방, 실시간 알림 흐름을 중심으로 만든 중고거래 백엔드 프로젝트

## Overview

TradeHub는 사용자가 상품을 등록하고, 다른 사용자가 예약을 걸고,  
거래 확정 이후 채팅과 알림 흐름으로 이어지는 **거래 중심 백엔드 서비스**입니다.

이 프로젝트는 단순 CRUD API를 나열하는 대신,

- 상품 등록과 조회
- 예약 생성과 확정
- 거래 완료 처리
- 채팅방 생성
- 실시간 알림 전송

까지 하나의 서비스 흐름 안에서 연결하는 데 초점을 두고 있습니다.

## Main Flow

TradeHub의 기본 흐름은 다음과 같습니다.

1. 사용자가 상품을 등록한다.
2. 다른 사용자가 해당 상품에 예약을 생성한다.
3. 판매자가 특정 예약을 확정한다.
4. 확정된 거래를 기준으로 후속 채팅 / 알림 흐름이 이어진다.
5. 거래 완료 요청을 통해 거래 상태를 마무리한다.

즉 이 프로젝트는 “상품 목록 서비스”가 아니라,  
**거래 상태가 변화하는 과정 전체를 다루는 서버**라고 볼 수 있습니다.

## What I Focused On

이 프로젝트에서 중요하게 본 지점은 다음과 같습니다.

- 상품, 예약, 거래 완료까지 이어지는 상태 변화 설계
- 거래 과정에서 필요한 채팅방 생성 흐름 연결
- 실시간 알림을 위한 SSE + Redis Pub/Sub 구조 시도
- 도메인별 패키지 분리와 계층 분리 연습

즉 단순 기능 구현보다,  
**거래 도메인을 여러 기능으로 쪼개지 않고 하나의 사용자 흐름으로 묶는 것**에 더 집중한 프로젝트입니다.

## Key Features

### Product

- 상품 생성 / 수정 / 삭제
- 단일 상품 조회
- 전체 상품 목록 조회
- 키워드 검색

### Reservation

- 예약 생성
- 상품별 예약 목록 조회
- 예약 확정
- 예약 수정 / 취소
- 사용자 기준 확정 예약 조회
- 거래 완료 요청

### Chat Room

- 거래 당사자 기준 채팅방 생성
- 전체 채팅방 조회
- 단일 채팅방 조회

### Notification

- SSE 기반 알림 연결
- 알림 생성 및 전송
- Redis Pub/Sub 기반 확장 구조

### User

- 사용자 생성 / 조회
- 금액 충전

## Project Structure

패키지는 도메인 기준으로 나뉘어 있습니다.

- `user`
- `product`
- `reservation`
- `chatroom`
- `notification`

각 도메인은 주로 다음 구조를 따릅니다.

- `controller`
- `service`
- `repository`
- `entity`
- `dto`

그리고 일부 도메인에는 다음 분리가 들어가 있습니다.

- `domain`
- `mapper`
- `port`
- `adapter`

이 구조를 통해 단순 컨트롤러-엔티티 직접 연결보다  
조금 더 역할이 나뉜 서비스 구조를 만들려고 한 흔적이 보입니다.

## Tech Stack

- Java 17
- Spring Boot 3.3
- Spring Web
- Spring Data JPA
- Validation
- Redis
- H2 Database
- Lombok

## What This Project Shows

TradeHub는 다음 역량을 보여주는 프로젝트입니다.

- 도메인을 나눠서 백엔드 API를 설계한 경험
- 상품 / 예약 / 거래 상태를 연결한 서비스 흐름 설계
- SSE + Redis Pub/Sub를 이용한 실시간 알림 구조 시도
- DTO, 엔티티, 서비스 계층을 분리해보려는 구조화 시도

즉 단순 게시판형 CRUD보다,  
**거래 도메인의 상태 변화와 사용자 상호작용을 백엔드 관점에서 묶어낸 프로젝트**로 볼 수 있습니다.

## One-line Summary

TradeHub는 **상품, 예약, 채팅, 알림을 하나의 거래 흐름으로 연결한 Spring Boot 기반 중고거래 백엔드 프로젝트**입니다.
